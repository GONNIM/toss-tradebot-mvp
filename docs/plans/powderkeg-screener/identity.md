## 🎯 정체성 (v2.0 · 2026-07-24 사용자 지시로 명시)

**목적**: **2026년 하반기 한국주식 투자 이익 창출** · 관찰 → 검증 → 매수/매도 판단.

**핵심**: 화약고 스크리너는 **투자 결정 지원 도구**. 다른 투자자보다 **정확한 정보로 정확한 매수/매도 시그널 생성**해 이익을 확보한다.

**여전히 금지**:
- 자동 매수·매도 실행 (반자동 티켓 §7-5 유지 · 1클릭 승인 필수)
- 검증되지 않은 가설 매매 (validated 게이트 통과만 티켓 생성)

**변경된 원칙 (v1.x → v2.0)**:
- v1.x: "예측 없음 · 관찰만 · 사용자 판단 근거 제공"
- v2.0: **"관찰하되 · 검증된 조건에서 매수/매도 판단으로 이익 창출"**
- Screener·backtest는 이제 **투자 시그널 생성 목적**으로 해석
- 조건이 tight해서 표본 부족하면 · 완화가 아니라 **역설계**로 실 데이터 기반 재정의 (P2-2c)

## 0. 요약 (Executive Summary)

- **자동 실행**: 이벤트 감지 (3분) · 트리거 처리 (1분) · 뉴스 폴링 (15분) · 보유 만료 (daily 08:00) — **4개 APScheduler 잡**
- **자동 실행 X (설계 원칙)**: **스크리너 자체 재평가 (`run_screener`)** — 자동 스케줄 등록 안 됨
- **사용자 수동 트리거**: 화약고 리스트 tier 판정 갱신은 UI '🔄 지금 재평가' 버튼 or `POST /screener/run`

## 1. 자동 실행되는 것 (APScheduler)

`backend/powderkeg/scheduler.py:register_powderkeg_jobs`:

| 잡 ID | 주기 | 역할 | 지시서 |
|---|---|---|---|
| `powderkeg_events_poll` | **3분** | DART 공시 폴링 · Type A/B 이벤트 자동 검출 | §7-3 |
| `powderkeg_triggers` | **1분** | 감지된 이벤트 처리 · 리스트 자동 제거 (Type B) · Telegram 알림 | §7-3 |

| `powderkeg_holding_expiry` | **매일 08:00 KST** | 보유 만료 검사 (반자동 티켓 만료 처리) | §7-5 |

**핵심**: 이 4개는 사용자가 아무것도 안 해도 자동 실행됨.

## 2. 자동 실행되지 않는 것 · 사용자 수동 트리거 필수

### 스크리너 재평가 (`run_screener`)
- **자동 스케줄 없음 · 설계 원칙**
- 이유: 재무·최대주주는 분기·연간 갱신 데이터 · 매일 재평가는 무의미하고 자원 낭비
- **수동 트리거 방법**:
  1. UI · '🔄 지금 재평가' 버튼
  2. API · `POST /api/v1/powderkeg/screener/run` (SNIPER_API_TOKEN 필요)

### 재무·최대주주·KRX 데이터 수집
- 자동 수집 없음
- **수동 트리거**:
  - `POST /collectors/dart-financials` (재무)
  - `POST /collectors/dart-shareholders` (최대주주)
  - `POST /collectors/krx-snapshot` (KRX)
- 통상 신 사업보고서 공시 (3월 말 · 8월 말 등) 이후 사용자가 대량 수집

## 3. 사용자 관점 오해 해소

### 우려: "지속적으로 타겟이 변하는 것 같다"

**실측 결과** (2026-07-22 기준):
```
최근 run_id 이력:
  2026-07-21 20:25  509 종목  ← 사용자 트리거
  2026-07-21 14:54    2 종목  ← 사용자 트리거
  2026-07-21 14:46  508 종목  ← 사용자 트리거
  2026-07-20 12:18  389 종목  ← 사용자 트리거
  2026-07-20 10:00  389 종목  ← 사용자 트리거
```
**모두 사용자가 명시 트리거한 결과** · 자동 재평가 없음.

### 왜 그렇게 보였나?
- 이번 세션에서 사용자 지시로 여러 번 재평가 실행 (P2 백필 → 재평가 → hotfix → 재재평가 → 다년 백필 → 재재재평가)
- 매 재평가마다 새 `run_id` 생성 · 리스트 tier·조건 갱신
- 그 사이 파싱 오류 hotfix (v1.31·v1.32) · 매핑 확장 (big_biz seed) · 금융업 배제 (P4-3) 등 로직 변경 반영

**즉 리스트가 변한 원인은 자동 재평가가 아니라 (a) 데이터 백필 (b) 로직 개선 후 사용자가 재평가한 결과**.

## 4. 사용자 표준 워크플로우

### 일상 흐름 (자동)
1. **APScheduler** · 3분마다 DART 공시 자동 폴링
2. Type A 이벤트 감지 (담보제공·상속·5% 보고 등) → **Telegram 알림**
3. Type B 이벤트 감지 (횡령·감사부적정·거래정지) → **자동 리스트 제거 + 알림**

### 사용자 개입 (수동)
1. 📱 **Telegram 알림 수신** — Type A 매수 후보 or Type B DO NOT TOUCH
2. 🌐 **UI 접속** — `optimus8.cafe24.com/powderkeg`
3. 🔄 **'지금 재평가' 클릭** (선택 · 최신 tier 확인용)
4. 🔍 **종목명 클릭** → 상세 팝업 · 재무·조건·이벤트·외부링크 확인
5. 🔒 **관찰용 lock** · 관심 종목 유지
6. 📊 **KRX 차트·네이버 금융·DART 공시** 확인 후 매수 판단
7. ⚙️ **반자동 티켓 생성** (validated=true 이벤트만)

### 정기 흐름 (분기·연간)
- 새 사업보고서 공시 (3월 말) 후:
  1. 재무 대량 재수집 (`POST /collectors/dart-financials · bsns_year=YYYY`)
  2. 최대주주 대량 재수집
  3. 감사의견 갱신
  4. 전체 재평가 · tier 재판정
  5. 신규 승격/강등 종목 확인 · lock 관리

## 5. 왜 자동 재평가를 하지 않는가? · 설계 근거

### 재무 데이터는 저빈도
- 사업보고서: 연 1회 (3~4월)
- 반기보고서: 연 1회 (8월)
- 분기보고서: 연 2회 (5·11월)
- 매일 재평가는 대부분 아무것도 안 바뀜

### 파싱·매핑 로직 변경 시 위험
- 이번 세션 예시: `_DEBT_KEYWORDS` 확장으로 서희 net_cash 40.6%→16.3% 강등
- 자동 재평가면 사용자 몰래 tier 뒤바뀜
- **수동 트리거 원칙**: 사용자가 로직 변경 인지 후 트리거 → provenance 명확

### Type B 즉시 대응은 별도 (자동)
- 횡령·감사부적정·거래정지는 실시간 대응 필수
- `powderkeg_triggers` 잡 (1분 주기)이 자동 처리 · 리스트 제거 + 즉시 알림
- 이 경우만 스크리너 우회하고 즉시 강등

## 6. 자동화 확장 옵션 (v2 검토)

**우려가 반복되면** 다음 옵션 검토 가능:

### 옵션 A · 리스트 종목만 미세조정 재평가 (일 1회 · 보수적)
- 매일 06:00 KST · **현재 리스트 (locked+user 종목)** 만 재평가
- KRX 시총·거래대금 변동 반영
- 유니버스 전체 재평가 X · 자원 절약
- **장점**: 사용자 몰래 tier 변화 발생 여부 감지
- **단점**: 신규 종목 승격 자동 감지는 여전히 X (사용자 지시 안 함 선택)

### 옵션 B · 전체 유니버스 자동 재평가 (일 1회 · 적극적)
- 매일 06:00 KST · 2,406 전종목 자동 재평가
- **장점**: 항상 최신 tier
- **단점**: 사용자 몰래 승격/강등 발생 · provenance 어려움 · 설계 원칙 위반

### 옵션 C · 안 함 (현재 · **사용자 지시**)
- 지시서 §7-3 자동 감시 = 이벤트만
- 스크리너 재평가는 사용자 판단 시 수동
- **장점**: 완전한 provenance · 로직 변경 안전
- **단점**: 신 재무 공시 즉시 반영 지연 (사용자가 재평가 안 하면)
