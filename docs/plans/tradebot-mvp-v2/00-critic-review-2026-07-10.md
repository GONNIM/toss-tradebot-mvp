## 0. 원 크리틱 요약 (재추적용)

외부 크리틱은 Toss Tradebot MVP의 4개 메뉴 각각에 대해 실전 매매 시 자금이 묶이거나 상투를 잡게 되는 치명적인 '갈증(고통)'을 지적하고, 이를 해결하기 위한 4가지 개선안을 제시함.

**지적된 4개 메뉴 갈증**
1. 섹터별 주도주 Top 3: 월간 수출 데이터의 후행성으로 인한 타임랙 및 진입 시점(Timing) 확보 불가.
2. 밈주 워치: 발견 시점에 이미 고점(상투). 실시간 호가, 분봉 거래량, 소셜 가속도 부재.
3. VIP 감시: 공시 시차(수일~수십 일) 및 장중 수급 집계 지연으로 인해 이미 15~20% 뜬 상태에서 추격 매수하는 리스크.
4. Activist Radar: 이벤트의 휘발성/장기화 판별 불가. 타임라인 및 법적 리스크 분석 부재로 베팅 규모 확대 제한.

**제안된 4개 개선안**
- ① Sector Leaders: 당일 거래대금 팩터(1,000억 + 20일 신고가) 실시간 가중치 결합.
- ② Meme Watch: 소셜 언급 가속도 지표 + 분봉 RSI 매칭 + Toss API 연동 트레일링 스톱.
- ③ VIP: 장중 거래원 분석으로 매집 실시간 추정 + 평균 매입단가 추정.
- ④ 공통: Toss API 자동매매 "원클릭 시드 20% 분할 매수" 버튼 도입.

## 2. 저자 개선안 판정 및 팩트 체크

### ① Sector Leaders — 거래대금 가중치 결합
- **판정**: 채택 가능 (매우 타당)
- **보완 및 주의사항**: 유료 KRX 데이터 라인 없이도 FinanceDataReader와 네이버 일봉 데이터를 활용해 구현 가능(Meme Watch의 인프라 재활용 가능). 단, 크리틱이 제안한 "20일 신고가"는 급격한 낙폭 과대 후 돌아서는 주도주 초기 국면을 놓칠 수 있으므로, **상대강도(RS, Relative Strength) 랭킹 시스템을 병행**하는 것이 유리함.

### ② Meme Watch — 소셜 가속도 + 분봉 RSI + 트레일링 스톱
- **소셜 가속도**: 이미 구현됨. 현재 `mention_velocity_30m` 지표가 작동 중이므로 크리틱의 "부재" 지적은 팩트 오류.
- **분봉 RSI**: 현재 시스템은 일봉 기준 데이터 팩터를 사용 중임. 분봉 도입은 API 호출 한도 및 데이터 소스(ApeWisdom 등)의 미커버 영역이 있어 **기술적 타당성 검토(Spike 조사) 선행 필요**.
- **트레일링 스톱**: 원천 불가. 아래 ④번 항목 참조.

### ③ VIP — 장중 실시간 거래원 분석 · 평균 매입단가 추정
- **장중 거래원 분석**: 현실적 인프라 제약으로 불가 (**기각**)
  - 이유: KRX 실시간 거래원(창구) 데이터는 고가의 유료 라이선스 및 증권사 데이터센터 피드가 필요함. 현재 MVP의 크롤링 환경(IP 차단 매트릭스 작동 중)에서는 장중 실시간 우회 수집이 원천 차단됨. 무료 소스로는 장 마감 후 요약 정보만 접근 가능.
- **평균 매입단가 추정**: 부분 채택 가능. 공시/필링 시점의 거래량 가중평균가격(VWAP)이나 필링 내 명시된 취득 단가 데이터를 파싱하여 **'세력 평단가' 배지** 형태로 제공 가능.

### ④ 공통 — Toss API 자동매매 원클릭 분할 매수 버튼
- **판정**: **채택 · Toss Open API v1.2.2 실전 확보 (2026-07-10 사용자 재확인)**
- **팩트 정정 (2026-07-10 세션 중 발견)**: 이전 판정에서 "토스증권 공식 자동매매 API는 개방되지 않음" 이라고 기술했으나 **오판**. 토스증권 Open API v1.2.2 가 정식 개방되어 있으며, 사용자가 이번 세션 중 클라이언트 등록 완료(WTS → 설정 > Open API). Base `https://openapi.tossinvest.com` · OAuth 2.0 Client Credentials · 계좌/주문 API에 `X-Tossinvest-Account` 헤더 · KR(KRX)+US 통합 · **조건주문 SINGLE/OCO/OTO 표준 지원**. 상세: `03-toss-openapi-integration.md` · 메모리 `reference_toss_open_api`.
- **방향성 재확인 (사용자 논거 반영, 2026-07-10)**:
  - **밈주/주도주 단타는 10~20초에 3~5% 급등락**. 알림 → 앱 켜기 → 호가 확인 → 매수까지 수십 초 딜레이 = 상투 확정.
  - 인간 심리 개입(뇌동매매)이 "실현손실 0" 대원칙을 파괴. 봇 결정은 **0.1초 이내 거래소 도달**해야 통계적 우위 확보 가능.
  - 즉, 원 크리틱의 "매매 실행 통합" 요구는 정확한 병목 진단이며, **핵심 병목은 브랜드가 아니라 실행 레이어의 부재 자체**임.
- **실전 아키텍처 — Broker Adapter Pattern + Toss 실전 채택**:
  ```
  [ 시그널 엔진 (Meme · VIP · Activist · Sector) ]
                    │
                    ▼
         [ Order Manager Interface (OMI) ]
         submit_order · cancel · get_balance · get_position
                    │
              ┌─────┴─────┐
              ▼           ▼
        [Paper Adapter] [Toss Adapter]
         시뮬 검증        실전 매매
                         (POST /api/v1/orders
                          + 조건주문 OCO)
  ```
  - **Toss Open API v1.2.2**: KR/US 통합 · REST 전용 (기존 FastAPI/APScheduler 정합) · 조건주문(SINGLE/OCO/OTO) 표준 지원. 사용자 API 키 확보로 즉시 구현 가능.
  - **Paper 어댑터를 Phase 1 필수**로 넣어 실 자본 투입 전 계약 통과 확보 (백테스트 인프라 재활용).
  - Broker Adapter Pattern 유지 이유: 향후 다른 증권사 API 지원 · 브로커 분산 실행 대비 옵션 보존.

## 3. 크리틱이 놓친 진짜 시스템 공백 (진짜 5%의 고통)

현재 코드베이스와 아키텍처를 대조했을 때, 외부 크리틱이 미처 발견하지 못한 실전 투자의 진짜 병목 구간은 다음과 같습니다.

1. **실제 매매 실행 레이어의 부재**: 시그널이 떠도 결국 텔레그램을 보고 사용자가 수동 주문을 넣어야 함. 자동매매 봇의 본질을 살리려면 **KIS Developers 계좌 연동 및 주문 모듈 통합**이 최우선임.
2. **포지션 사이징 및 리스크 예산(Risk Budget) 룰 부재**: Kelly Criterion이나 변동성 타게팅(Vol Targeting), 혹은 종목별 Max Drawdown 캡(Cap) 로직이 없음. "절대 실현 손실 0"이라는 목표를 달성하기 위한 **자금 관리 수학이 빠져 있음**.
3. **크로스-메뉴 백테스트 인프라 불균형**: Meme Watch는 백테스트 이력이 존재하나, Sector Leaders 및 Activist Radar는 과거 데이터 기반의 시뮬레이션 인프라가 없어 시그널의 누적 승률을 신뢰하기 어려움.
4. **다중 시그널 병합(Multi-hit) 뷰 부재**: 특정 종목(예: WEN)이 밈주에도 걸리고, VIP 매집에도 걸리고, 행동주의 타겟에도 동시에 히트했을 때 이를 **'슈퍼 시그널'로 승격**시켜 주는 통합 스코어링 시스템이 없음.
5. **알림 피로도(Alert Fatigue) 관리 미흡**: 시그널 폭발 시 실시간 알림 피로감을 제어할 개인 자산 규모별, 투자 성향별 필터링 프로파일이 부재함.
6. **1분봉 실시간 진입 타점 부재**: 5분 폴링이 최소 단위. 즉시 진입은 여전히 사용자 판단 몫.

## 4. 최종 판정 요약

| 구분 | 평가 결과 | 비고 |
|---|---|---|
| 저자 지적 4개 | **3/4 공감 · 1/4 부분 공감** | 투자자 관점의 문제 진단 및 갈증 포착은 매우 예리함. |
| 저자 제안 4개 | **1/4 채택 · 1/4 기구현 · 1/4 제약 · 1/4 부분 채택+재설계** | ④번은 매개체는 오해이나 병목 진단(실행 레이어 부재)은 정확 — Broker Adapter Pattern으로 재설계 수용. |
| 시스템 실제 공백 | **6건의 치명적 갭 발견** | 매매 실행 채널, 리스크 관리 알고리즘 등 봇의 본질적 공백 확인. |

**총평**: 본 크리틱은 실전 관점에서 유의미한 갈증을 짚어냈으나, 기술적 실현 가능성(KRX 데이터 라이선스 등)과 브로커리지 API 환경에 대한 이해 부족으로 인해 제안을 그대로 수용할 수는 없습니다. 따라서 **"지적한 고통(고충)은 전적으로 수용하되, 이를 해결하는 기술적 매체와 실행 방안은 전면 재설계"**하는 방향이 타당합니다.

## 5. 결정 — 트랙 C 확정 (2026-07-10 사용자 승인)

**최종 결정**: **트랙 C (통합 로드맵)** 채택. 트랙 A(현실화·매매 실행 채널)와 트랙 B(내실화·리스크·백테스트) 를 결합하여 단계별 릴리즈로 진행.

### 5-1. 확정 근거
1. **매매 실행 부재는 P0**: 실전 트레이딩의 병목이자 "실현손실 0" 원칙의 붕괴 지점 — 트랙 A 없이는 모든 시그널이 알림에서 멈춤.
2. **안전장치 없이 실 매매 금지**: 트랙 B의 리스크 예산·Paper 어댑터·백테스트가 실 자본 투입 전 선행되어야 함.
3. **아키텍처는 브로커 중립**: KIS를 우선 채택하되 토스 API 향후 개방을 대비해 Broker Adapter Pattern 도입 — 상위 시그널 엔진은 브로커에 종속되지 않음.

### 5-2. Phase 요약 (상세: `01-track-c-roadmap.md` · 2026-07-10 Toss 전환 반영)
| Phase | 스코프 | 기간 | 실 자본 리스크 |
|---|---|---|---|
| **Phase 0** | 🟢 Toss Open API 클라이언트 등록 (사용자 완료) · `.env` 값 저장 · 허용 IP 등록 | 준비 | 없음 |
| **Phase 1** | OMI 인터페이스 + Paper 어댑터 + 리스크 예산 + Kill Switch | 2주 | 없음 (시뮬레이션만) |
| **Phase 2** | TossAdapter 실 연동 + Smoke Test 6단계 + 실계좌 소액 5거래일 forward test | 3주 | 10만 원 상한 하드코딩 |
| **Phase 3** | 다중 시그널 병합 · 조건주문(OCO) 익절·손절 자동 세팅 · 크로스-메뉴 백테스트 | 3주 | 없음 |
| **Phase 4** | Toss REST 폴링 최적화 (1분봉) + Sector RS 랭킹 + Meme 분봉 RSI Spike 조사 | 2주 | 확대 검토 |
| **Phase 5** | KR/US 통합 실행 검증 + 조건주문 확장(재진입·분할 매도) | 2주+ | 정상 운용 |
